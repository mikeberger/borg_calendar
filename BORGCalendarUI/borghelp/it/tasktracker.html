La Lista Attività

La lista delle attività è usata per tenere traccia di tutte le attività definite con una data di inizio e di fine, con differenti stati, descrizioni e soluzioni.

E' l'utente a decidere se qualcosa è un'attività oppure un regolare appuntamento. Gli appuntamenti possono essere usati per delle semplici operazioni che non richiedono di conservarne traccia, come per es. pagare una scadenza, buttare l'immondizia, ecc. Le Attività sono qualcosa di più lungo, più complesso, come per es. "creare il codice di un nuovo programma", "progettare un nuovo elemento", "ristrutturare il pavimento del bagno", ecc. - cioè qualcosa che abbia più operazioni successive da controllare, che richiedano una documentazione di una soluzione e possibili slittamenti della data del termine.

La finestra principale delle Attività mostra una lista scorrevole di attività, o compiti (task). Il menu Azione può essere usato per aggiungere, aggiornare e chiudere un'attività. I tasti a scelta opzionale (radio-botton) possono essere usati per mostrare solo attività "aperte", "chiuse", fatte ma ancora senza "Rapporto di Prestazione" (cioè in attesa di un rapporto sulle prestazioni), oppure "Tutte".

Una stringa per filtrare le attività può essere inserita nel campo "Filtro". Quando si clicca su "Filtro" solo le attività che contengono il testo inserito saranno visualizzate.

Le Attività possono essere filtrate anche secondo la loro categoria, usando il menu a tendina "Categoria".

Le Attività con Date di consegna (o termine) saranno mostrate nel calendario..

La lista delle attività è anche progettata per tenerne traccia successivamente per registrarle in una lista di lavori completati. Questo è chiamato Performance Review (Rapporto di Prestazione). Speriamo che la maggior parte degli utenti di BORG non debba trattare queste infernali burocrazie. Una delle cose per le quali BORG può essere usato è un consuntivo del lavoro svolto in un certo intervallo di tempo, per es. un anno - e assicurarsi così di poter rivendicare dei crediti per alcuni lavori già svolti, separandoli da lavori non svolti. Alcuni utenti potranno non identificarsi in queste problematiche, ma altri probabilmente potranno trarre beneficio da queste funzionalità. Probabilmente nel futuro verrà rilasciata un'opzione che permetterà di rendere invisibile questa funzionalità a chi non la volesse utilizzare.

Una volta aggiunta, un'attività segue una serie di stati configurabili dall'utente di BORG.

Se non vi piacciono i valori di default per Tipi e Stati, potete modificarli. Leggete la sezione successiva per modificare Tipi e Stati della Atività per ulteriori informazioni. Gli stati delle attività di default sono quelli utilizzati dagli autori di questo programma, e forniscono solo un esempio di cosa può essere fatto. Non ci sono problemi se volete eliminarli e sostituirli con altri vostri.


Nel modificare le attività, molte funzionalità presenti nella finestra sono ovvie. Comunque:

  • il numero di persone assegnate (n. Pers.) e Priorità non fanno altro che riferirsi all'attività. Usatele come preferite.

Sotto-Attività

Per ogni attività, 5 sotto-attività possono essere definite dall'utente. Queste 5 caselle possono essere usate per tener traccia di qualsiasi cosa necessaria all'utente per avere cura di ogni parte dell'attività complessiva. Per esempio, se un'attività è chiedere un finanziamento per ristrutturare la vostra casa, sotto-attività possono essere "chiedere una valutazione", "fax delle ricevute di pagamento", "spedizione della richiesta", "chiusura finale", "libro pagamenti". Le sotto-attività devono essere controllate solo nella propria attività, non dipendono da nessun'altra.

Ogni Tipo di Attività può anche avere 5 tipi-specifici di sotto-attività che sono in comune a tutte le attività di un particolare tipo. I tipi-specifici di sotto-attività possono essere modificati dall'utente (dalla Release 1.0). Vedere seguito.

 

    • le finestrelle standard di BORG (Sotto-Attività) per attività di tipo CODE e DOC forse non sono applicabili alla maggior parte degli utenti (soprattutto quelli che non sono degli sviluppatori di codice). Ora spieghiamo il senso delle finestrelle create per il tipo CODE:
      • Delta = scrittura di codice con controllo del sorgente
      • una data con una traccia blu che è stata riempita -- guardate la ISO 9000 per uteriori dettagli -- o Dilbert per ulteriori dettagli sulla ISO 9000
      • MR = una richiesta di modifica è stata pubblicata (richiesta di modifica, o comunque il vostro committente probabilmente richiede di riferire su caratteristiche/problemi del codice)
      • Code Insp = ispezione del codice eseguita.

    • qui sono spiegate le sottoattività del tipo DOC:
      • DocNum = un numero ufficiale di documento è stato stabilito
      • VGs = alcuni elementi grafici sono stati realizzati per la presentazione in un meeting
      • Blueprint (vedi sopra)
      • MR (vedi sopra)
      • Library = il documento è stato ufficialmente archiviato in quache libreria.


Evidentemente, le sotto-attività presenti sono molto specifiche del lavoro svolto dall'autore di questo programma. Probabilmente il set predefinito di attività e sotto-attività dovrebbe essere più generico o facile da personalizzare.

Modifica di Tipi Attività e Stati

La lista di Attività di default definite sopra sono probabilmente adatte per alcuni utenti ma inadatte per altri. L'utente ha la possibilità di modificare la lista di attività sia per Tipo che per Stato. Per modificare Tipi e Stati di Attività, prima di tutto dovete usare, dal menu delle attività, l'opzione per esportare le informazioni sui Tipi di attività in un file XML. Salvate questo file nel caso vogliate tornare indietro.

Il formato di questo file evidentemente è XML - ma un XML semplificato. Il semplice parser XML usato in BORG non si occupa di commenti, attributi, prologo - quindi il file dovrebbe semplicemente contenere gli elementi XML descritti in seguito.


Evidentemente, qualche genere di GUI dovrebbe essere disponibile per personalizzare il compito di editare il file XML. Questa GUI non è al momento disponibile.

Ad ogni modo, il file XML contiene un elemento radice (root), sotto il quale ci sono tutti i tipi di attività. Per aggiungere un nuovo Tipo di attività dovete aggiungere semplicemente un nuovo elemento sotto l'elemento radice.

Dentro ogni tipo di elemento, sono descritte le transizioni di stato permesse. Per ogni stato consentito, c'è un elemento che dovrebbe contenere un elemento vuoto per ogni stato che può essere raggiunto a quello stato. Per ogni tipo possono essere presenti fino a 5 elementi. Ci sono le caselle dei tipi-specifici.

Questo è un esempio di ramificazione per il tipo di attività BORG_REL usato dall'autore:

<BORG_REL>
  <OPEN>
    <IN_PRG/>
  </OPEN>
  <IN_PRG>
    <CLOSED />
  </IN_PRG>
  <CLOSED>
    </IN_PRG>
  </CLOSED>
  <Web Start JAR>
  <File Release>
  <Update latest_version>
  <Tag CVS>
  <News>
</BORG_REL>

Il tipo di attività è BORG_REL. L'attivita puà avere lo stato OPEN, IN_PRG, CLOSED. Ogni stato contiene la lista di stati a cui si puograve; giungere. Per esempio, per lo stato OPEN, l'unico stato successivo può essere IN_PRG. Gli elementi CB mostrano le sotto-attività che appaiono per ogni attività BORG_REL.

Ogni tipo di attività parte dallo stato OPEN. Questo è una imposizione di BORG. Ogni Tipo di Attività può essere forzato in uno stato CLOSED tramite il menu Azione del programma, ma questo non può essere permesso manipolando il file XML.

Una volta che avete creato il file XML con i Tipi di attività, dovrete importarlo nel programma usando l'opzione di importazione nel menu Opzioni. Questo deve contenere TUTTI i vostri stati, non solo alcuni nuovi che volete aggiungere. Utilizzate l'opzione di esportazione degli stati esistenti per apprendere a creare un file XML con tutti gli stati che desiderate.

Borg non importerà un file XML non valido; dovrete quindi correggere ogni errore nella struttura del file XML.

Se il documento XML è valido, ma non corretto, potrete orreggere il file XML ed importarlo nuovamente. Nessun danno sarà fatto da errori logici nel file XML, a meno che è probabile che non siate capaci a generare correttamente le vostre Attività.
